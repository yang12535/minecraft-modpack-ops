## 定义包类型意图

在选择文件之前先确定包类型。

- **公开玩家包：** 面向玩家或朋友分发的工件。经过隐私审查后应可分享。
- **私有完整恢复包：** 包含凭证或部署材料，供运维者恢复服务。将其视为机密材料。
- **仅游戏归档：** 仅包含 Minecraft 服务端目录或指定的游戏文件。
- **根路径相对恢复包：** 保留 `root/server`、`etc/systemd/system`、`etc/frp`、`etc/minecraft`、`usr/local/bin` 等路径，使兼容主机可从 `/` 解压恢复。

如果用户说“公开”“给玩家”“给朋友分发”，默认使用公开模型。如果用户说“私有”“灾备”“完整恢复”“包含密钥”“解压即可部署”，默认使用私有完整恢复模型。

## 检查归档前状态

停止或打包前：

1. 通过 RCON 或服务端控制台查询在线玩家
2. 当用户说“没人玩就处理”时，检查近期加入/离开日志
3. 服务运行中且即将执行破坏性操作时，先运行 `save-all flush`
4. 需要一致世界快照时，通过正常生命周期命令停止
5. 确认 Java 和隧道进程/监听器处于预期最终状态
6. 检查源主机和目标主机的剩余磁盘空间

不要从安静的终端推断“没人在线”。使用 Minecraft 在线人数或日志。

## 调整备份保留

备份模组可能在配置看似合规的情况下吃掉小型 VPS 的大部分磁盘。

至少检查：

- 备份类型：完整备份还是仅修改内容备份
- 间隔或定时器
- 数量上限，例如 `backupsToKeep`
- 大小上限，例如 `maxDiskSize`
- 无玩家在线时是否仍会运行备份
- 是否按世界创建子目录，因为限制可能按子目录生效
- 输出路径

在 30-50 GB 的小磁盘上，完整备份策略通常同时需要较低的数量上限和总大小上限。私人朋友服的一个实际起点是保留 3-4 个完整备份，并将备份总空间控制在约 4-5 GB，再按实际世界大小和磁盘预算调整。

清理已有备份时：

1. 列出备份的时间、大小和路径
2. 除非用户指定恢复点，否则保留最新文件
3. 仅删除属于目标服务器的备份归档
4. 重新检查目录大小和磁盘剩余空间
5. 编辑保留策略前，保留一份旧备份配置

不要将常规备份目录放入公开包。只有当私有恢复包需要保留回滚历史时，才包含保留的备份。

## 构建公开玩家包

即使只分享给朋友，也应把公开包视为可发布工件。

通常包含：

- `mods`、`config`、`defaultconfigs`、`kubejs`、数据包、资源包、自定义资产、库、加载器文件、运行脚本和预期的 `world`
- 复现整合包所需的服务端清单或安装器元数据
- 已脱敏的 `server.properties`

通常排除：

- `logs`、`crash-reports`、备份目录、调试转储、缓存和安装器生成日志
- `ops.json`、`whitelist.json`、`banned-players.json` 和 `banned-ips.json`
- `usercache.json`、用户名缓存、玩家来源 IP 日志和原始 UUID 映射笔记
- RCON 配置、FRP 配置、FRP 令牌、SSH 密钥、云厂商元数据和 systemd 运维单元
- 包含代理地址、凭证、私有端点或一次性部署状态的本地辅助脚本

加入归档前先脱敏配置：

- 清空 `rcon.password`
- 除非公开包明确说明如何设置新密码，否则禁用 RCON
- 清空 `server-ip`
- 将 Web 或配置 UI 密码替换为占位符
- 移除私有代理设置和私有 API 令牌
- 保留有意设置的玩法配置，例如难度、在线模式、视距、模拟距离、玩家上限和整合包特定配置

优先创建临时脱敏覆盖文件并将其加入 tar 流。不要为了构建公开工件而直接修改正在使用的服务端配置。

传输前执行路径级和内容级审查。搜索以下名称：

```text
password token secret rcon frpc frps proxy private_key ops.json usercache whitelist banned logs simplebackups crash-reports
```

将关键词命中视为审查提示，而不是自动排除规则。

## 构建私有完整恢复包

私有恢复归档可以包含凭证，但必须在用户明确要求私有包或完整恢复包之后才这样做。

有用内容包括：

- 完整 Minecraft 服务端目录
- 保留的备份归档，如果交付目标包含回滚历史
- `mcstart-*`、`mcstop-*` 和 RCON 辅助命令等生命周期脚本
- systemd 单元和环境文件
- RCON 密码文件
- 该部署所需的 FRPC 配置、FRPS 配置、令牌文件和隧道二进制
- 归档内的一份简短恢复说明，写明预期解压路径和恢复后检查项

对根路径相对恢复包，尽量保留所有权和权限：

```sh
tar --xattrs --acls -cpf - root/server etc/frp etc/minecraft etc/systemd/system usr/local/bin
```

在另一台主机恢复后，验证：

- 服务用户和组存在
- 文件所有权和权限仍能保护令牌和 RCON 配置
- 已运行 `systemctl daemon-reload`
- 提供商 NAT、公共端点和防火墙规则已匹配新主机
- 全局启动/停止命令仍指向恢复后的服务端路径

不要把私有归档清单、令牌哈希、密码或端点机密粘贴到公开 issue 或仓库中。

## 规划压缩资源

`tar.zst` 是 Linux 到 Linux 归档工作的良好默认值。高压缩设置例如：

```sh
zstd --ultra -22 --long=31 -T0
```

可能使用大量 CPU 核心和内存。2 GB 窗口配合多工作线程可能超过 8 GB 内存，并可能需要临时交换空间。这是预期行为，不是 Minecraft 特有问题。

使用激进压缩前：

- 检查输出归档和临时交换所需的剩余磁盘空间
- 检查可用内存和交换空间
- 判断机器是否空闲到可以承受 CPU 和 I/O 饱和
- 监控 OOM kill
- 压缩后移除临时交换空间

如果主机内存紧张，先减少线程，再降低压缩等级。如果主机磁盘紧张，在更大的机器上压缩，或将 tar 输出流式传输到另一台主机。

已经压缩过的备份 zip、jar、图片和区域数据通常不会明显缩小。归档小于预期时，常见原因是有意排除了备份、日志或缓存；通过比较源目录大小和清单计数来验证，而不是猜测。

## 验证归档

验证应分层进行：

1. 在归档旁生成 SHA-256 文件
2. 运行解压测试，例如 `zstd -t --long=31 archive.tar.zst`
3. 使用 `tar -tf` 生成清单
4. 检查必需路径存在
5. 对公开包检查禁止路径不存在
6. 从归档流中检查敏感文件，而不仅检查实时文件系统
7. 当包体积可疑时，比较世界区域文件数量
8. 复制后验证目标端哈希一致

对于世界，比较源和归档中 `.mca` 等区域文件的数量。公开包体积本身不能证明世界被截断或完整。

## 传输与存储

使用能保留字节并报告失败的工具传输归档，例如 `scp`、`rsync` 或已验证的对象存储上传。传输后验证哈希。

私有恢复包不要放入：

- 公开仓库
- issue 跟踪器
- 共享文档目录
- 普通玩家分发目录
- 日志或聊天记录

公开包通过隐私审查后可以更自由分发，但仍可能包含玩家建筑、坐标、玩家数据和世界历史。除非用户明确希望公开世界，否则将世界视为敏感数据。

## 故障排查

- **归档异常小：** 按目录比较源大小，确认备份/日志是否被有意排除，并比较世界区域文件数量。
- **归档异常大：** 检查备份目录、日志、崩溃报告、缓存和已压缩资产。
- **压缩被杀死：** 检查 OOM 日志，添加临时交换，减少 `zstd` 线程，或流式传输到更大的机器。
- **压缩时磁盘写满：** 删除半成品归档，避免本地临时交换，排除预期的大目录，或在另一台主机上压缩。
- **公开归档包含机密：** 删除工件，从脱敏覆盖重建；如果包已离开可信环境，轮换暴露凭证。
- **私有恢复包解压后无法运行：** 检查服务用户、文件权限、systemd reload、Java 版本、提供商映射和路径假设。

## 完成报告

报告：

- 包类型：公开、私有、仅游戏或完整恢复
- 最终归档路径、大小和哈希状态
- 备份保留变更以及剩余备份数量/大小
- 服务器和隧道最终保持运行还是停止
- 已通过的验证阶段：解压、清单、必需路径、禁止路径、传输后哈希
- 未解决的恢复要求，例如提供商 NAT 重建或缺失服务用户

除非用户在授权环境内明确要求，否则不要报告凭证、令牌哈希、玩家 IP、原始 UUID 映射或私有端点细节。
