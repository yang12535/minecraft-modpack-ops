# 模组包部署与运维

## 目录

- 运行环境
- 整合包检查
- 下载与加载器安装
- 客户端模组修剪
- 运行时管理
- 分层验证
- 容量评估
- 故障排查

## 运行环境

在操作之前先了解：

- 源存档或整合包标识符
- Minecraft 和确切的加载器版本
- 目标主机、安装目录、端口、日志和全局命令名称
- Java 版本、架构、CPU、内存、磁盘和交换空间
- 现有的 Minecraft 进程和冲突的监听器
- 当前服务风格和健康的本地脚本约定
- 下载限制、代理地址和并发限制
- 玩家上限、视距、模拟距离和内存上限
- 是否有匹配的客户端可用

使用不同的目录、PID 文件、日志、游戏端口和 RCON 端口。不要停止或覆盖无关的服务器。

## 检查整合包

从实际工件开始：

1. 在解压前列出存档内容。
2. 读取清单以获取 Minecraft、加载器、项目/文件 ID、哈希、侧、覆盖和捆绑 JAR 信息。
3. 清点配置、默认配置、脚本、数据包、资源包、着色器、地图和模板。
4. 建立文件名、模组 ID、版本、依赖、侧、来源和哈希的对照表。
5. 当清单不完整时检查 JAR 元数据。

将平台侧元数据视为提示。一个视觉导向的模组仍可能注册了必需的服务器网络通道或存根。

## 构建服务器整合包

在实际可行时使用支持清单的导入器，然后验证其输出。

保留：

- 精确的游戏和库版本
- 服务器相关配置
- KubeJS、CraftTweaker 或等效脚本
- 数据包和世界生成数据
- 官方安装器生成的加载器参数和库

排除用户状态工件，如日志、缓存、截图、启动器元数据、崩溃报告和个人按键绑定，除非有意需要。

从官方分发版安装加载器。确认 Java 与特定加载器和整合包的兼容性。

## 可靠下载

- 仅对下载器或安装器进程应用代理设置。
- 在游戏启动前移除代理变量和 JVM 代理标志。
- 将并发视为受代理带宽、远程节流、磁盘 I/O 和完整性检查制约的预算。
- 并行处理独立的文件；不要给一个小文件分配过多线程。
- 使用有界退避重试临时故障。
- 保留已验证的文件。
- 当元数据提供 SHA-1 或 SHA-256 时进行验证。

当平台导入器拒绝受限项目时，使用精确的项目元数据和授权的回退方式。CurseForge ForgeCDN 常用格式：

```text
https://edge.forgecdn.net/files/<file-id-div-1000>/<file-id-mod-1000-padded>/<url-encoded-filename>
```

验证当前的元数据和结果哈希。`403` 可能意味着文件 ID 或文件名有误。

启动后，检查 Java 进程环境，确认仅用于下载的代理设置已移除。

## 保守修剪客户端模组

目标是最小化必要修剪。

- **已证明客户端专用：** 渲染器、着色器、菜单、GUI 动画、地图、HUD、准星、客户端音频、按键绑定或视觉覆盖，且有确认的元数据或行为。
- **不确定：** 库、背包辅助、信息模组、音乐集成、配方查看器插件、优化模组以及任何声明为 `both` 的模组。
- **服务器/内容：** 世界生成、生物、机器、魔法、维度、结构、存储、网络和依赖项。

初始阶段仅移除已证明为客户端专用的候选项。

使用：

- 清单和 JAR 侧声明
- 依赖关系图
- 专用服务器启动行为
- 无效的 `CLIENT`/`DEDICATED_SERVER` 堆栈跟踪
- 已注册的网络通道
- 使用完全匹配的客户端整合包进行真实登录

重要故障模式：

- `mismatched mod channel list`（模组通道列表不匹配）：恢复确切列出的模组和客户端版本。信息类或音乐模组仍可能需要服务器网络通道。
- 客户端类在专用服务器上加载：识别有问题的 JAR；仅当保留的模组不需要它时才移除。

记录每个排除项：文件名、版本、证据和测试结果。

## 运行时管理

审查：

- 绑定地址和游戏端口
- 玩家上限
- 视距和模拟距离
- 在线模式和白名单
- 操作员权限级别
- RCON 状态和端口
- 堆内存限制和主机保留

`view-distance` 控制实际发送/加载的服务器区块。它与模拟距离和客户端远距离地形渲染不同。

为操作系统、原生/JVM 开销、文件系统缓存和管理服务留出内存。

### 启动命令

创建一个全局启动命令，能够：

- 解析固定目录
- 防止重复启动
- 使用加载器生成的参数
- 通过 systemd 或可靠的 `setsid`/`nohup` 分离
- 写入 PID 文件和持久化日志
- 迅速返回

单独轮询就绪状态。不要为整个服务器生命周期保持 SSH 命令打开。

### 主机启动与进程监管

服务器需要随主机启动时，优先使用专用 systemd 单元，而不是 Shell 配置文件、cron 或面板登录钩子。

服务模型必须与实际进程行为一致：

- 条件允许时，直接运行 Java 并使用 `Type=simple`
- 只有在保留会将 Java 放入后台的现有启动脚本时，才使用 `Type=forking` 和准确的 `PIDFile`
- 不要将后台化启动脚本与 `Type=simple` 组合；systemd 可能跟踪包装脚本而不是 Java

配置：

- 启动还管理隧道时，使用 `After=` 和 `Wants=` 等待网络就绪
- 有界的启动和停止超时
- `Restart=on-failure` 和适度的重试延迟
- 执行优雅 RCON 流程的明确 `ExecStop`
- 在正常的多用户启动目标下启用

验证：

- `systemd-analyze verify` 对新单元不报告错误
- 单元已启用，并能通过真实停止/启动或重启
- `MainPID` 是预期的 Java 进程
- PID 文件、进程、监听器和单元状态一致
- 稳态运行期间重启计数不会增长
- 主机启动后，无需交互式 Shell 即可到达 Minecraft 就绪状态

不要仅凭 `enabled` 推断开机启动成功。已启用的单元仍可能失败、跟踪错误进程，或在所需存储和网络就绪前启动。

### 停止命令

尝试按以下顺序：

1. 通过 RCON 执行 `save-all flush`
2. 通过 RCON 执行 `stop`
3. 使用有界轮询等待进程优雅退出
4. 若进程仍未退出，发送 `SIGTERM`
5. 仅在迫不得已时强制终止

在发送信号之前验证 PID 是否属于此服务器。

### RCON

- 在服务器上生成强密码
- 存储在仅 root 可读的文件中
- 将监听器限制在可信管理路径
- 在有用时提供仅 root 可用的辅助工具
- 绝不在报告中打印密码

使用 RCON 进行保存、停止、TPS、玩家管理和 OP 授予。对于 OP 级别四，同时设置 `op-permission-level=4` 和玩家条目。

## 分层验证

### 静态

- 所需文件存在且哈希匹配
- 加载器和 Java 版本兼容
- 权限和配置路径正确
- 端口无冲突

### 启动

- 进程保持存活
- 预期的监听器出现
- 日志到达 `Done (...)`
- 没有致命的加载器、依赖、注册表、Mixin 或分发失败残留

将配方、标签和进度警告与启动阻塞分开分类。

### 网络

- 从服务器进程命名空间外部测试
- 确认绑定和防火墙行为
- 使用玩家将使用的确切地址

### 客户端握手

- 使用预期的客户端加入
- 精确解决缺失模组和通道不匹配
- 区分登录兼容性和后续世界加载失败
- 到达世界

这在客户端模组修剪后是强制性的。

### 生命周期

- 在破坏性测试前检查在线玩家数量
- 玩家在线时推迟完整停止/重启测试，除非用户明确接受中断
- 通过 RCON/全局命令停止
- 确认世界和维度保存
- 确认进程/监听器消失
- 再次启动
- 到达第二次 `Done (...)`
- 确认无重复进程

### 运行时

- Java 中无代理残留
- 在探索和区块生成期间观察 TPS/MSPT
- 检查内存、GC 和交换
- 检查备份、日志、OP、白名单和权限

日志可能包含玩家名称、UUID、IP 地址、聊天、坐标和命令输出。将完整日志保存在授权主机上。当诊断需要摘录时，在放入持久工件或公开报告之前，对无关身份和地址进行脱敏处理。

## 容量评估

不要凭一张空闲状态的 `htop` 截图来估算容量。

- `htop` 可能将 Java 线程显示为多行。
- 主线程时间、区块生成、GC 和内存通常在总 CPU 达到 100% 之前就已经限制了服务器。
- 更高的视距会成倍增加内存和网络成本。
- 模拟距离会增加 tick 成本。
- 分开探索的玩家比站在一起的玩家更重。

使用 TPS/MSPT、常驻内存、GC 和预期的最坏情况工作负载。将估算值表述为带假设的范围。

## 故障排查

- `mismatched mod channel list`：在排查防火墙/网络之前先对齐命名的模组。
- 缺失依赖：检查每个模组的故障块和元数据。
- 无效客户端类：识别 JAR 及其保留的依赖项。
- 手动下载失败：更换下载方法，保留确切的 ID 和哈希。
- SSH 在启动时挂起：分离服务并使用有界轮询。
- 停止超时：使用 RCON，然后正常信号；检查保存日志。
- OOM/交换/GC 暂停：检查堆和距离，然后考虑增加虚拟机内存。
- 空闲 TPS 良好但探索时卡顿：分析世界生成和区块加载。

## 防护栏

- 不要删除源整合包或无关服务器。
- 不要仅仅为了降低数量而移除不确定的模组。
- 不要将下载代理持久化到运行时。
- 不要使用前台进程作为服务管理器。
- 在保存和优雅停止尝试之前不要强制终止。
- 没有客户端握手就不要声称玩家兼容性，或明确说明尚未验证。
- 不要将模组包存档、服务器日志、世界、玩家缓存、凭证文件或生成的启动机密提交到文档仓库。
