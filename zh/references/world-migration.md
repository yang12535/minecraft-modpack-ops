# 世界替换与 UUID 迁移

## 目录

- 存档检查
- UUID 模型选择
- 完整数据转换
- 分阶段替换
- 世界默认值和 OP 记录
- 验证
- 防护栏

## 视作数据迁移

世界替换不仅仅是复制一个目录。玩家身份可能出现在文件名、文本存储、独立 NBT、区域/实体 NBT 以及模组特定数据库中。

## 检查存档

在触碰运行中的世界之前：

1. 验证校验和并测试每个 ZIP 成员。
2. 拒绝绝对路径、目录遍历和意外的链接。
3. 检测直接的世界根目录相对于一个包装目录。
4. 拒绝模糊的多重根目录。
5. 确认 `level.dat`。
6. 清点以玩家为键的数据。

至少检查：

- `playerdata`、进度和统计
- GameStages 和装饰盔甲
- FTB Quests 和 FTB Teams
- 经济、领地、队伍、传送石、背包和入门套件追踪
- 记分板和持久化模组存储
- 实体和方块实体内的所有权

不要假设 UUID 仅出现在文件名中。

## 选择一种 UUID 模型

从最终的 `online-mode` 推导目标模型。

### 在线模式

使用当前规范玩家名称的 Mojang 认证 UUID。查询权威的个人资料数据并记录证据。

### 离线模式

对精确的大小写敏感字节计算 Java 名称 UUID：

```text
OfflinePlayer:<player-name>
```

这是一个 MD5 版本 3 UUID。付费账户在加入 `online-mode=false` 服务器时也会收到此离线 UUID。

构建显式映射表：

```text
玩家名称 | 源 UUID | 目标 UUID | 源模式 | 目标模式 | 证据
```

保留玩家将使用的确切名称大小写。保持此表为临时性，或放在授权的管理环境中。默认仅报告映射数量，不报告原始玩家名称和 UUID。

## 转换所有玩家引用

仅重命名 `playerdata/<uuid>.dat` 或更改 `ops.json` 是不够的。

转换：

- 以 UUID 为键的文件名和目录
- 带连字符和不带连字符的文本 UUID
- NBT `IntArray[4]` UUID
- NBT `Most`/`Least` 长整型对
- 所有者、投掷者、受信任玩家、领地、队伍、团队、任务、经济和通知引用
- `.mca` 文件中的实体和方块实体所有权

使用结构化的 JSON/TOML/SNBT/NBT 解析器。使用支持 NBT 的区域写入器，而非二进制搜索替换。

重写区域文件时：

- 保留每个区块的压缩类型
- 在可行时保留时间戳
- 正确重建位置表
- 保持未修改区块在语义上不变
- 拒绝外部或不支持的区块格式，除非有意处理

对副本运行两次转换。第二次必须报告：

- 零 UUID 值变化
- 零路径重命名
- 零区域区块重写

此幂等性检查是一个强烈信号，表明旧 UUID 形式未被遗漏。

## 通过分阶段目录替换

1. 优雅停止 Minecraft。
2. 确认 Java 和游戏/RCON 监听器已消失。
3. 解压到同一文件系统上的分阶段目录。
4. 在分阶段目录中转换 UUID 和世界默认值。
5. 验证 `level.dat`、预期的玩家文件、计数和目标 UUID。
6. 移除过时的 `session.lock`。
7. 应用用户的保留决定：
   - 当需要或有价值时保留带时间戳的备份
   - 仅当用户明确表示旧世界可丢弃时才直接删除
8. 将分阶段目录重命名为配置的 `level-name`。

使用同一文件系统的原子重命名（如 `os.replace`）进行最终移动。绝不要解压到运行中或活动世界上。

## 设置认证和默认值

保持以下一致：

- `online-mode`
- `enforce-secure-profile`
- 白名单策略
- `op-permission-level`
- 难度
- 极限模式
- 游戏规则，如 `keepInventory`

导入的 `level.dat` 可能携带覆盖 `server.properties` 假设的难度和游戏规则。需要时使用解析器修改世界 NBT。

通过 RCON 验证运行时值：

```text
difficulty
gamerule keepInventory
```

## 正确构建 OP 级别四

对于每个已知的目标玩家：

- 使用目标模式 UUID
- 将名称和 UUID 写入 `ops.json`
- 设置 `"level": 4`
- 设置 `op-permission-level=4`
- 一致地更新 `usercache.json`
- 启动后运行幂等的 `op <名称>`

期望的响应是该玩家已经是操作员。

离线模式允许用户名冒充。在公网暴露之前，需要可信网络、严格白名单或兼容的认证层。

将源世界、转换后的世界、个人资料响应、`ops.json`、`usercache.json`、日志、坐标、背包、领地和经济记录视为私有玩家数据。不要将它们复制到技能仓库中，也不要在验证后保留临时提取目录。

## 验证迁移后的世界

### 静态

- ZIP 通过完整性检查
- 世界根目录明确无歧义
- 所有预期的目标 UUID 文件存在
- 旧的 UUID 文件名不存在
- 映射是幂等的

### 启动

- 服务器到达 `Done (...)`
- 没有破坏性的世界、注册表或玩家数据加载失败出现
- 预期的维度加载

过时的配方或进度警告可以是非致命的；单独分类。

### 运行时

- RCON 报告预期的难度和游戏规则
- `save-all flush` 成功
- OP 命令是幂等的
- 日志显示登录时的目标 UUID

### 玩家验收

使用代表性转换账户加入并验证：

- 背包和选中的快捷栏槽位
- 末影箱
- 位置和维度
- 进度和统计
- 任务和团队
- 经济和领地
- 宠物、坐骑、背包和拥有的实体

当有匹配客户端可用时，没有真实玩家登录就不要声明身份迁移已完全验收。

## 故障排查

- 登录后背包为空：目标模式和玩家数据 UUID 不一致。
- 原版数据保留但模组数据重置：UUID 保留在模组特定文件名或存储中。
- 宠物或领地失去所有权：嵌入的 NBT 或区域数据未被转换。
- OP 条目被忽略：`ops.json` UUID 使用了错误的认证模式或权限级别低于四。
- 服务器重新生成世界：分阶段根目录或 `level-name` 有误。
- `session.lock` 错误：从源复制了过时的锁。

## 防护栏

- 选择目标模式后，绝不要混用在线和离线 UUID。
- 绝不要在离线模式下从 Mojang 个人资料推断付费玩家的 UUID。
- 绝不要使用原始二进制替换执行 UUID 迁移。
- 绝不要在 Minecraft 运行时覆盖活动世界。
- 除非用户明确允许，绝不要丢弃旧世界。
- 绝不要在未解释名称冒充风险的情况下公开暴露离线模式服务器。
- 绝不要在公开报告中包含原始 UUID 映射、玩家背包、坐标、个人资料属性或完整的身份承载日志。

## 完成报告

通用地报告源世界、目标认证模式、玩家映射数量、转换的数据类别、备份/删除决定、难度/游戏规则、OP 结果、启动结果和玩家登录验收状态。仅在授权请求者需要时披露玩家名称或 UUID。不要打印凭证或私有个人资料属性。
