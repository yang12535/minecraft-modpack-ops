# 稳定性与多出口可观测性

## 目录

- 观测范围
- 节点生命周期
- 分层健康检查
- 历史事件统计
- 受控主动测试
- Minecraft 运行时稳定性
- 多出口对比
- 故障排查

## 定义观测窗口

在统计事件之前，说明精确的开始和结束时间、时区以及窗口的缘由。

确定：

- 哪些 Minecraft 进程生命周期在范围内
- 哪些 FRPS 和 FRPC 服务预期处于活跃状态
- 每个中继何时创建、停止、替换或销毁
- 配置或提供商 NAT 映射是否发生变化
- 是否发生了测试探测、上传、备份或维护
- 时钟和日志时间戳是否对齐

不要将活跃中继与在窗口期间已停止或销毁的中继进行比较。在生命周期变化处分割窗口，或将已退役的中继标记为不可比较。

## 首先验证节点生命周期

在解释故障之前：

1. 在可用时确认实例在提供商控制面板中仍然存在
2. 确认其预期的公共地址和 NAT 映射是最新的
3. 检查启动时间和服务启动时间
4. 识别有意的关机、删除、替换或过期的计费
5. 标注有效的退役时间

连接拒绝、超时、SSH Banner 停滞和无尽的 FRPC 重试在中继退役后是预期行为。不要将这些样本归因于运营商质量或 FRP 可靠性。

## 区分分层健康

将每一层视为独立的证据：

1. 提供商实例存在
2. 公共 NAT 边缘接受流量
3. 中继操作系统响应
4. FRPS 正在监听并已认证
5. FRPC 已建立控制会话
6. 代理已注册且健康
7. 本地 Minecraft 监听器就绪
8. 公共 Minecraft 状态协议成功
9. 匹配的客户端到达世界

以下信号单独来看是不充分的：

- `systemctl active`：重试中的 FRPC 进程可以无限期保持活跃
- 进程存在：它可能已断开连接或卡住
- ICMP 成功：游戏或控制映射仍可能损坏
- TCP 连接：提供商 NAT 边缘可能在没有可用后端的情况下接受握手
- 过时的已建立套接字：旧的工作连接可能在控制健康丢失后仍然存在

在称某个出口健康之前，需要最近的 FRPC 登录成功、代理注册、健康检查成功和应用层 Minecraft 响应。

## 仔细统计历史事件

将被动历史统计与主动测试分开。

对于每次统计，记录：

- 源日志和主机
- 观测窗口和时区
- 使用的事件模式
- 样本数量
- 成功和失败的定义
- 排除项和已知盲点

有用的 FRP 事件包括：

- FRPC 控制连接尝试
- 成功登录
- 认证或拨号错误
- 代理启动和关闭事件
- 健康检查转换
- FRPS 接受的连接
- 工作连接故障
- 服务启动和重启

有用的 Minecraft 事件包括：

- 加入
- 登录阶段超时
- 运行时超时
- 干净的客户端断开连接
- 踢出和握手失败
- 服务器重启和崩溃

不要假设 FRPS 接受的连接等同于成功玩家登录。状态探测、扫描器和失败的握手也会产生连接。FRPC 后面的 Minecraft 可能看到代理地址而非玩家源地址，因此后端日志可能无法识别玩家使用了哪个公共出口。

避免重复统计：

- 一次超时可能同时产生 `lost connection` 和 `left the game`
- 一次主动探测可能出现在 FRPS 和 Minecraft 日志中
- 服务重试循环可能产生重复尝试而没有独立的用户流量

## 运行受控主动测试

对每个活跃出口使用相同的探测实现、源主机、时机、超时和样本数量。

优先使用 Minecraft 状态协议而非原始 TCP。报告：

- 尝试次数
- 成功和失败次数
- 成功率
- 平均、最小、最大和 P95 延迟
- 协议或响应不匹配

运行足够的样本以暴露间歇性故障，但限制速率以避免影响游戏。标记主动测试流量，以免之后被误认为被动历史。

当玩家在线时，避免破坏性的生命周期测试。优先使用状态探测、只读服务检查和日志分析。

## 检查 Minecraft 运行时稳定性

检查：

- 进程运行时间和重启次数
- 当前和峰值常驻内存
- 配置的堆内存和实际堆占用
- Young、Concurrent 和 Full GC 行为
- 主机可用内存和交换使用
- 代表性探索下的 TPS 或 MSPT
- `Can't keep up` 频率和最差延迟
- OOM、看门狗、崩溃和保存失败记录
- 磁盘空闲空间和世界增长
- 加入、超时和断开连接模式

不要仅凭 Linux `free` 内存低就推断内存泄漏。比较进程 RSS、GC 后的堆占用、原生开销、页面缓存和交换随时间变化的趋势。

没有 Full GC 或 OOM 并不意味着内存余量充足。一台几乎无可用内存且交换不断增长的主机仍可运行，但恢复余量很差。仅在考虑原生 JVM 和操作系统预留后，才建议增加主机内存、减小堆或降低距离。

将启动或区块生成期间的孤立 `Can't keep up` 记录与持续高 MSPT 区分对待。报告次数、最差延迟、工作负载上下文以及该状况是否持续。

## 比较多出口

以共同窗口为每个出口构建一行：

| 字段 | 含义 |
| --- | --- |
| 生命周期 | 活跃、已停止、已退役、已销毁或未知 |
| FRPS 状态 | 监听器、启动时间、重启 |
| FRPC 状态 | 最近登录、重试、健康状态 |
| 被动连接 | 接受或观察到的连接（排除主动生成探测） |
| 被动故障 | 同一层内的显式错误 |
| 主动状态探测 | 成功、失败、延迟摘要 |
| 玩家证据 | 可归因于该出口的加入、超时或报告 |
| 可信度 | 已确认、推断或不可用 |

绝不要将已销毁或其他非活跃出口与活跃出口进行比较。将其标记为排除并解释原因。

避免合并不可比的数字。示例：

- FRPS 接受的连接不能与 Minecraft 加入直接比较。
- FRPC 重试失败不是玩家连接失败。
- 主动探测不能添加到被动历史总数中。

## 故障排查

- 进程活跃但反复拨号错误：FRPC 在重试循环中存活；检查 FRPS、提供商 NAT 和中继生命周期。
- 公共 TCP 连接但 SSH Banner 或 Minecraft 状态超时：NAT 边缘可能在无响应后端的情况下接受流量。
- ICMP 正常但 FRP 控制被拒绝：检查映射的控制端口、FRPS 监听器、防火墙和实例生命周期。
- 状态成功但玩家无法加入：检查模组通道、在线模式、UUID 处理、登录阶段日志和客户端版本。
- 超时在上传期间集中出现：关联路由器排队、上行饱和、重传和缓冲区膨胀。
- 可用内存接近零且交换增长：检查 GC 后的 RSS 和堆，然后保留操作系统/原生余量。
- 大量警告但游戏症状很少：按子系统分类警告，而非使用原始警告数量作为稳定性指标。

## 完成报告

报告：

- 精确的观测窗口和时区
- 每个被比较节点的生命周期状态
- 被动和主动结果分别呈现
- 成功/失败定义和排除项
- Minecraft 重启、超时、延迟、内存、GC、交换和磁盘发现
- 出口归属的可信度级别
- 未解决的缺口，包括缺失的提供商历史或不可用日志

对玩家身份、源 IP、凭证、UUID 和不必要的原始日志行进行脱敏处理。
