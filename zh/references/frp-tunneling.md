# FRP 与 NAT 暴露

## 目录

- 网络路径映射
- 二进制文件获取
- FRPS 加固
- FRPC 配置
- 启停耦合
- 验证
- 故障排查

## 映射每一跳

在配置之前先记录路径：

```text
玩家 -> 公共游戏端点 -> 提供商 NAT -> frps 远程端口
frpc -> 公共控制端点 -> 提供商 NAT -> frps 绑定端口
frpc -> 本地 Minecraft 地址和端口
```

提供商 NAT 端口和 FRP 端口是分开的层。提供商可能将公共控制端口映射到 VPS 的 `7000`，将公共游戏端口映射到 VPS 的 `25565`。

确定：

- 中继操作系统和架构
- 内部 FRPS 绑定端口
- FRPS 远程游戏端口
- 提供商公共映射
- 本地 Minecraft 地址和端口
- 防火墙和监听器冲突
- 从 FRPC 到公共控制端点的出站可达性

FRP TCP 代理不需要启用 IP 转发。

## 可靠获取 FRP

- 确定当前官方版本
- 下载精确架构的构建
- 验证发布的校验和
- 当中继没有互联网出口时，从可信机器上传二进制文件
- 记录两侧安装的版本

不要从任意镜像安装未验证的二进制文件。

## 加固 FRPS

以非特权 systemd 服务运行 FRPS。

配置：

- 固定的绑定地址和控制端口
- 基于文件的随机令牌认证
- 强制 TLS
- `allowPorts` 限制为所需的 Minecraft 远程端口
- 低单客户端端口限制
- 有界日志保留
- 除非明确需要，否则禁用仪表板

保护：

- 令牌文件：模式 `0600`
- 配置：root 拥有，仅在需要时组可读
- 日志目录：仅服务账户可写

使用 systemd 加固，例如：

- `NoNewPrivileges=true`
- `PrivateTmp=true`
- `ProtectHome=true`
- `ProtectSystem=strict`
- 狭窄的 `ReadWritePaths`

使 FRPS 在中继重启后保持启用。中继服务是基础设施，通常即使在 Minecraft 停止时也保持活跃。

## 配置 FRPC

以独立的非特权账户和 systemd 服务运行 FRPC。

设置：

- 公共 FRPS 地址
- 提供商映射的控制端口
- 相同的基于文件的令牌字节
- TLS
- 本地 Minecraft 绑定地址和端口
- FRPS 远程游戏端口
- 针对 Minecraft 的 TCP 健康检查
- 有界日志保留

健康检查可防止 FRPS 在本地游戏监听器存在之前就通告代理。

当用户需要 FRPC 跟随 Minecraft 启停命令时，保持 FRPC 禁用独立启动激活。

## 将 FRPC 耦合到 Minecraft

保留现有的 PID、RCON、日志和分离行为。

备份脚本，精准编辑，运行 `bash -n`，然后安装。

启动命令应：

- 启动或检测 Minecraft
- 在进程存在后启动 FRPC
- 如果 Minecraft 已在运行但隧道缺失，则修复 FRPC
- 报告隧道故障而不丢失本地服务器控制
- 打印本地和公共端点，不打印凭证

停止命令应：

- 优雅保存并停止 Minecraft
- 在游戏退出后停止 FRPC
- 即使 Minecraft 已不存在也停止 FRPC
- 在中继上保持 FRPS 运行

不要让 FRPC 的进程占用 SSH 调用。

## 安全传输凭证

- 在服务器上生成令牌
- 通过 SSH 直接流式传输到另一台服务器
- 不要写入本地工作区
- 不要回显到聊天或日志
- 处理凭证前禁用 Shell 追踪
- 避免在命令参数或环境转储中传递凭证
- 在授权会话中随后比较 SHA-256 哈希
- 确认文件具有相同的字节，包括行尾

认证成功一次不是忽略字节不匹配的理由；规范化它以避免版本相关的解析行为。
不要将令牌或其哈希复制到完成报告、仓库、问题跟踪器或其他持久工件中。

## 验证四种状态

### 仅 FRPS

- FRPS 服务活跃
- 控制端口监听
- Minecraft 代理端口不监听

### Minecraft 加 FRPC

- FRPC 成功登录
- 健康检查变为成功
- 代理启动
- FRPS 打开远程游戏端口

### 公共端点

使用 Minecraft 状态协议或真实客户端。验证：

- 协议/版本
- 玩家数量和上限
- MOTD
- 合理的响应延迟

纯粹的 TCP 连接是不够的。提供商 NAT 可能在后端不存在时接受 TCP 握手。

### 生命周期

在无玩家在线时：

1. 运行全局停止命令
2. 确认所有维度保存
3. 确认 FRPC 变为非活跃
4. 确认 FRPS 关闭游戏代理端口
5. 确认 Minecraft 状态在公网失败
6. 运行全局启动命令
7. 使用有界轮询等待
8. 确认 RCON 就绪
9. 确认 FRPC 健康成功
10. 确认公共 Minecraft 状态恢复

将服务器保持在请求的最终状态。

## 故障排查

- 控制公共端口不可达：检查提供商 NAT、出站过滤和 FRPS 绑定监听器。
- FRPC 认证失败：比较令牌哈希和 TLS/认证设置。
- FRPC 登录但代理被拒绝：检查 `allowPorts`、端口限制和现有监听器冲突。
- 代理延迟启动：检查本地 TCP 健康检查和 Minecraft 启动时间。
- 公共 TCP 连接但状态超时：NAT 边缘可达但 FRPS 代理或后端缺失。
- FRPS 远程端口在 FRPC 停止后仍保持：检查过时的客户端/控制状态和服务日志。
- FRPC 在启动时意外启动：禁用 systemd 启用并保留显式命令耦合。
- Minecraft 停止但隧道保持：确保停止脚本中的每个提前返回路径都调用 FRPC 停止辅助函数。

## 防护栏

- 除非明确需要且严格限制，否则不要通过 FRP 暴露 RCON。
- 不要默认暴露 FRPS 仪表板。
- 当一个游戏端口足够时，不要允许任意远程端口。
- 没有具体需要时不要以 root 运行 FRPS 或 FRPC。
- 不要将令牌直接放在全局可读的 TOML 中。
- 不要仅从中继本身验证；测试面向玩家的公共端点。
- 将中继 IP、提供商映射、客户端源地址和 FRP 日志视为敏感基础设施数据。仅向授权请求者披露确切值。

## 完成报告

报告 FRP 版本、中继和本地服务状态、映射状态、TLS 和令牌文件认证、健康检查行为、全局命令耦合以及公共 Minecraft 状态结果。仅在授权请求者需要时披露确切端点。绝不要报告令牌或其哈希。
