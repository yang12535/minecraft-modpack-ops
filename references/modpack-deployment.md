## Operating Context

Discover before acting:

- source archive or pack identifier
- Minecraft and exact loader versions
- target host, install directory, ports, logs, and global command names
- Java versions, architecture, CPU, memory, disk, and swap
- existing Minecraft processes and conflicting listeners
- current service style and healthy local script conventions
- download restrictions, proxy address, and concurrency limits
- player cap, view distance, simulation distance, and memory ceiling
- whether a matching client is available

Use distinct directories, PID files, logs, game ports, and RCON ports. Do not stop or overwrite an unrelated server.

## Inspect The Pack

Start from the actual artifact:

1. List the archive before extraction.
2. Read the manifest for Minecraft, loader, project/file IDs, hashes, sides, overrides, and bundled JARs.
3. Inventory configs, default configs, scripts, datapacks, resource packs, shaders, maps, and templates.
4. Build a table of filename, mod ID, version, dependencies, side, source, and hash.
5. Inspect JAR metadata when the manifest is incomplete.

Treat platform side metadata as a hint. A visually oriented mod can still register a required network channel or server stub.

## Build The Server Pack

Use a manifest-aware importer when practical, then verify its output.

Preserve:

- exact gameplay and library versions
- server-relevant configs
- KubeJS, CraftTweaker, or equivalent scripts
- datapacks and world-generation data
- loader arguments and libraries generated by the official installer

Exclude user-state artifacts such as logs, caches, screenshots, launcher metadata, crash reports, and personal keybinds unless intentionally required.

Install the loader from its official distribution. Confirm Java compatibility for the specific loader and pack.

## Download Reliably

- Apply proxy settings only to downloader or installer processes.
- Remove proxy variables and JVM proxy flags before game startup.
- Treat concurrency as a budget governed by proxy bandwidth, remote throttling, disk I/O, and integrity checks.
- Parallelize independent files; do not assign excessive threads to one small file.
- Retry transient failures with bounded backoff.
- Preserve already verified files.
- Verify SHA-1 or SHA-256 when metadata provides it.

When a platform importer refuses restricted projects, use exact project metadata and an authorized fallback. CurseForge ForgeCDN has commonly used:

```text
https://edge.forgecdn.net/files/<file-id-div-1000>/<file-id-mod-1000-padded>/<url-encoded-filename>
```

Verify current metadata and the resulting hash. A `403` may mean the file ID or filename is wrong.

After startup, inspect the Java process environment and confirm download-only proxy settings are absent.

## Prune Client-Only Mods Conservatively

The goal is minimal necessary pruning.

- **Proven client-only:** renderer, shader, menu, GUI animation, map, HUD, crosshair, client audio, keybind, or visual overlay with confirming metadata or behavior.
- **Uncertain:** libraries, inventory helpers, information mods, music integration, recipe-viewer addons, optimization mods, and anything declaring `both`.
- **Server/content:** world generation, mobs, machines, magic, dimensions, structures, storage, networking, and dependencies.

Remove only proven client-only candidates initially.

Use:

- manifest and JAR side declarations
- dependency graphs
- dedicated-server startup behavior
- invalid `CLIENT`/`DEDICATED_SERVER` stack traces
- registered network channels
- a real login with the exact client pack

Important failure patterns:

- `mismatched mod channel list`: restore the exact listed mods and client versions. Informational or music mods may still require server network channels.
- client class loaded on a dedicated server: identify the offending JAR; remove it only if retained mods do not require it.

Record every exclusion with filename, version, evidence, and test result.

## Runtime Administration

Review:

- bind address and game port
- player cap
- view distance and simulation distance
- online mode and whitelist
- operator permission level
- RCON state and port
- heap limits and host reserve

`view-distance` controls real server chunks sent/loaded. It differs from simulation distance and client-side distant terrain rendering.

Leave memory for the OS, native/JVM overhead, filesystem cache, and management services.

### Start Command

Create a global start command that:

- resolves a fixed directory
- prevents duplicate starts
- uses loader-generated arguments
- detaches via systemd or reliable `setsid`/`nohup`
- writes a PID file and persistent log
- returns promptly

Poll readiness separately. Do not hold the SSH command open for the full server lifetime.

### Stop Command

Prefer:

1. `save-all flush`
2. `stop`
3. wait for clean exit
4. normal termination signal if RCON fails
5. forced termination only as a last resort

Verify a PID belongs to this server before signaling it.

### RCON

- generate a strong password on the server
- store it in a root-only file
- restrict the listener to trusted management paths
- provide a root-only helper when useful
- never print the password in reports

Use RCON for saves, stopping, TPS, player administration, and OP grants. For OP level four, set `op-permission-level=4` as well as the player entry.

## Validate In Layers

### Static

- required files exist and hashes match
- loader and Java versions are compatible
- permissions and configuration paths are correct
- ports do not conflict

### Boot

- process remains alive
- expected listener appears
- log reaches `Done (...)`
- no fatal loader, dependency, registry, mixin, or distribution failure remains

Classify recipe, tag, and advancement warnings separately from startup blockers.

### Network

- test from outside the server process namespace
- confirm bind and firewall behavior
- use the exact address players will use

### Client Handshake

- join with the intended client
- resolve missing mods and channel mismatches exactly
- distinguish login compatibility from later world-load failures
- reach the world

This is mandatory after client-only pruning.

### Lifecycle

- stop through RCON/global command
- confirm world and dimension saves
- confirm processes/listeners disappear
- start again
- reach a second `Done (...)`
- confirm no duplicate process

### Runtime

- proxy absent from Java
- TPS/MSPT observed during exploration and chunk generation
- memory, GC, and swap inspected
- backups, logs, OPs, whitelist, and permissions checked

Logs can contain player names, UUIDs, IP addresses, chat, coordinates, and command output. Keep full logs on the authorized host. When diagnosis requires an excerpt, redact unrelated identities and addresses before placing it in a persistent artifact or public report.

## Evaluate Capacity

Do not estimate capacity from one idle `htop` screenshot.

- `htop` may show Java threads as many rows.
- Main-thread time, chunk generation, GC, and memory often limit the server before aggregate CPU reaches 100%.
- Higher view distance multiplies memory and network cost.
- Simulation distance adds ticking cost.
- Players exploring separately are heavier than players standing together.

Use TPS/MSPT, resident memory, GC, and the expected worst-case workload. State estimates as ranges with assumptions.

## Failure Routing

- `mismatched mod channel list`: align named mods before investigating firewall/network.
- missing dependency: inspect per-mod failure blocks and metadata.
- invalid client class: identify the JAR and retained dependents.
- manual-download failure: change download method, preserving exact IDs and hashes.
- SSH hangs on startup: detach service and use bounded polling.
- stop timeout: use RCON, then normal signal; inspect saving logs.
- OOM/swap/GC pauses: review heap and distances, then consider increasing VM memory.
- exploration lag with good idle TPS: profile world generation and chunk loading.

## Guardrails

- Do not delete the source pack or unrelated servers.
- Do not remove uncertain mods merely to lower the count.
- Do not persist a download proxy into runtime.
- Do not use a foreground process as the service manager.
- Do not force-kill before save and graceful stop attempts.
- Do not claim player compatibility without a client handshake, or state clearly that it remains unverified.
- Do not commit modpack archives, server logs, worlds, player caches, credential files, or generated launch secrets to a documentation repository.

## Completion Report

Report installation directory, loader, Java, ports, RCON, command names, heap, player/distance settings, exclusions with evidence, download verification, and each validation stage.
